4. Reflection on the Validation Process

Running my pages through the W3C Markup Validation Service was a useful final step that caught a handful of small issues I would not have spotted by simply previewing the pages in a browser. On the feedback page, the validator flagged a stray unclosed <div> inside the reviews grid that had been left over from an earlier layout experiment. Because the browser had silently auto-closed it in a way that still rendered correctly, I had not noticed it during development. Correcting it also clarified the nesting structure of the reviews section.

On the team page, the validator noted a warning about a redundant aria-label on the <nav> element that duplicated the text already present in the landmark role. While this is a warning rather than an error, I kept it in place because the explicit label improves discoverability in screen readers that list navigation landmarks by name. The content page validated cleanly on the first run, which I attribute to using semantic HTML5 elements throughout and ensuring all id attributes on anchor targets were unique and descriptive. The main lesson was that validation should be run early and often during development, not only at the end — small structural issues are much easier to trace and fix when the page is still being built than after extensive styling has been added.
