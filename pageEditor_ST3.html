2. Technical Description of Implemented Pages

Feedback & Programme Directory Page

➚ Live Page ➚ Validation

HTML Structure

The page is built around a <main> landmark containing three clearly separated <section> elements: an introductory panel, the scrollable programme directory, and the community reviews and feedback form. Each section uses aria-labelledby to associate its heading, providing meaningful landmarks for screen readers. The six programme entries are each wrapped in an <article> tag since each card is a self-contained unit of content with its own heading, image, tags, and call-to-action. The feedback form uses nested <fieldset> elements with <legend> labels — "Personal Details" and "Feedback Details" — so that inputs are announced in meaningful context by assistive technologies. A live-region <div id="formStatus" aria-live="polite"> at the base of the form announces validation outcomes without requiring focus to move.

CSS Techniques

The programme card grid uses grid-template-columns: repeat(2, minmax(0, 1fr)), giving two equal columns that collapse to one below 960px via a @media query. The scrollable directory container is constrained with max-height: 700px and overflow-y: auto, with a custom ::-webkit-scrollbar gradient thumb in cyan-to-gold to make the scroll affordance feel intentional. The reviews and form area uses a two-column grid with grid-template-columns: 1.1fr 1fr, giving the review strip slightly more visual weight. Each review card uses a border-left: 4px solid var(--accent-cyan) accent to reinforce hierarchy between quoted content and the form. Input focus states use a branded box-shadow: 0 0 0 3px rgba(0, 240, 255, 0.12) instead of the default browser outline, remaining WCAG-compliant while matching the team's aesthetic.

Design & Implementation Rationale

Using nested <fieldset> and <legend> instead of plain <div> wrappers was deliberate — these elements convey semantic grouping so that screen readers announce the group name when focus enters, making "Full Name" read as "Personal Details — Full Name". The form uses novalidate to suppress browser-native error bubbles and replace them with JavaScript-driven inline errors styled consistently with the team's design. The <datalist> element on the Interest Area field provides native autocomplete without any external library, keeping the page lightweight and standards-compliant.

Team Page

➚ Live Page ➚ Validation

HTML Structure

The Team page uses two <section> elements inside <main>: a centred intro and the team grid, which is labelled aria-label="Team Members" so screen readers identify it as a distinct region. Each of the four member cards is a <div> with tabindex="0" making it keyboard-focusable. Inside each card, the photograph uses a descriptive alt attribute identifying the person by name and team role. The hover overlay is an absolutely positioned <div class="extra-info"> containing structured paragraphs for name, role, pages, and a short description, which surfaces only on hover or keyboard focus.

CSS Techniques

The card grid uses grid-template-columns: repeat(2, 1fr) with a fixed max-width: 850px and margin: 0 auto to keep the two-by-two layout centred — preventing cards from stretching awkwardly on wide viewports. Each card uses CSS's padding-box and border-box background-origin syntax to apply a gradient border via the background shorthand, producing a gold-to-cyan border without a wrapper element. The overlay starts at opacity: 0 and transitions to opacity: 1 over 0.3s on parent :hover or :focus. A transform: translateY(-6px) on card hover provides the lift effect consistent with the team's interactive card pattern.

Design & Implementation Rationale

The hover-overlay pattern keeps the default card view clean — a portrait, name, and role at a glance — while rewarding interaction with richer detail. Adding tabindex="0" combined with :focus-visible styling ensures keyboard users can trigger the overlay identically to mouse users, with a gold outline appearing only on keyboard focus rather than mouse click. The grid-template-columns: 1fr breakpoint at 768px stacks cards vertically on mobile so every member's information remains readable without horizontal overflow.

Content Page: Access to Justice

➚ Live Page ➚ Validation

HTML Structure

All article content is placed inside a single <article> element, reflecting that this is one self-contained piece of long-form writing. Within it, four <section> elements carry unique id attributes — "meaning", "barriers", "digital", "action" — that the Table of Contents links to via href="#id" internal anchors. The Table of Contents itself is wrapped in a <nav aria-label="Internal Navigation"> element, distinguishing it from the site header navigation in the accessibility tree. A highlight box near the top surfaces the three most important takeaways before the reader commits to the full article. Pull-quote blocks use a styled <div class="quote-panel"> with italic text to visually separate key statements from body copy. Each content image carries a descriptive alt attribute tied to its editorial context.

CSS Techniques

The page sets scroll-behavior: smooth and scroll-padding-top: 110px on html so that clicking any Table of Contents link scrolls the section into view smoothly and positions it below the fixed header. The main container uses max-width: 1000px with backdrop-filter: blur(16px) to maintain the team's glass-card aesthetic. Section headings are styled in var(--accent-cyan) with a border-bottom: 1px dashed separator, providing visual variety from the gold accents used elsewhere. Content images apply a transition on transform and border-color to scale and highlight on hover without disrupting reading flow. The page also includes a custom CSS-only back-to-top button styled as a circular anchor with a gold border, which is positioned fixed in the lower-right corner.

Design & Implementation Rationale

Wrapping the Table of Contents in a <nav> landmark was motivated by accessibility — having two distinct navigation regions lets screen reader users jump between site-wide links and in-page links using the landmarks menu without re-traversing the full page. The scroll-padding-top fix on the html element addresses a common usability problem: without it, the fixed header visually covers the target section heading after any anchor click. Each section id value was chosen to be descriptive and meaningful (e.g. id="barriers") rather than generic, improving both readability and navigation when the page is traversed by heading structure in a screen reader.

3. Challenges & Lessons Learned

Challenge 1: Custom Form Validation Without a Backend

My initial approach relied on HTML5 required attributes alone. This blocked empty submissions but the browser's native error bubbles were visually inconsistent across browsers and clashed with the team's dark-glass design. I solved this by adding novalidate to suppress native bubbles and writing custom JavaScript validation instead. Each field is checked with element.value.trim() on submit; if invalid, a styled error is injected into a dedicated .error-text element below the field and the border turns red via inline style. The success confirmation only renders when isValid === true, preventing a bug I had earlier where the success message appeared even when errors were present. This taught me that HTML5 validation is a good starting point but JavaScript is needed for a fully controlled, branded user experience.

Challenge 2: Hover Overlay Hidden Behind the Card Image

On the Team page, the .extra-info overlay was not fully appearing over the card on hover — the member photograph was painting on top of it and cutting off the text. The issue was a CSS stacking context problem. The absolutely positioned overlay needed a z-index: 2, and I confirmed the card wrapper had position: relative and overflow: hidden set to create the correct stacking context. This was a practical lesson in how z-index only works within an established stacking context — simply assigning a high number to an element is not enough if the parent does not define the context.

Challenge 3: Anchor Links Scrolling Behind the Fixed Header

On the content page, clicking a Table of Contents link scrolled correctly but the fixed site header covered the section heading on arrival, making it invisible until the user scrolled up manually. My first attempt was to add padding-top to each section, but this created unwanted visual gaps throughout the page. The clean solution was setting scroll-padding-top: 110px on the html element — a single CSS declaration that offsets all anchor scroll targets by the header height simultaneously, with no JavaScript needed.

Challenge 4: CSS Conflict with the Shared Team Template

When I merged my pages into the shared repository, a local button selector in my feedback page CSS was overriding the global button styles in Student 1's style.css, affecting the navigation header buttons site-wide. Rather than using !important — which would have created new override problems for teammates — I scoped my local button rules using more specific selectors such as .feedback-form .action-btn. This resolved the conflict cleanly and reinforced the importance of narrow, targeted selectors in a shared codebase.

Challenge 5: Programme Grid Orphaned Card on Mid-Width Screens

The two-column programme grid worked well on desktop and stacked cleanly on mobile, but on a medium-width tablet the layout produced a single orphaned card on the last row that stretched to full width, looking visually broken. I adjusted the grid to use minmax(0, 1fr) on both columns and set a max-width on the containing section, which constrained the grid to a size where two cards always fit comfortably without one stretching alone. This highlighted that grid behaviour at breakpoints requires careful testing at intermediate viewport widths, not just the extremes.

4. Accessibility Considerations

Accessibility was treated as a design requirement throughout development rather than an afterthought. The following specific measures were applied across my three pages:

  • Semantic Form Markup: All form inputs use explicit <label> elements linked via for and id pairing. Nested <fieldset> and <legend> group related inputs so screen readers announce the group context alongside the field name. The form's aria-live="polite" status box announces validation outcomes without shifting focus away from the form.
  • Keyboard Navigation: The full feedback form is navigable by Tab and Enter only, tested manually without a mouse. All interactive elements — form fields, buttons, team cards — are reachable and operable via keyboard. Custom focus rings are clearly visible, meeting WCAG 2.1 SC 2.4.7 (Focus Visible). Team cards use :focus-visible to show a gold outline on keyboard focus while avoiding it on mouse interaction.
  • Image Alt Text: Every team member photo has a descriptive alt attribute identifying the person by name and role. Content page images describe their editorial context rather than appearing generic. Decorative background elements use pointer-events: none and are kept out of the focus order.
  • Internal Navigation Landmark: The Table of Contents on the content page is wrapped in <nav aria-label="Internal Navigation">, creating a separate landmark from the site header. This allows screen reader users to jump directly to in-page links via the landmarks menu without re-traversing the full page structure.
  • Colour Contrast: All text across my pages uses gold-on-dark and light-grey-on-dark combinations that exceed the WCAG AA minimum contrast ratio of 4.5:1 for normal text.

5. Compliance with JANET Regulations

All three pages I authored comply fully with the JANET Acceptable Use Policy and the university's responsible web publishing guidelines:

  • Appropriate Content: The Feedback page collects only voluntarily provided opinions about the website. The Programme Directory is fictional and educational, directly aligned with SDG 16 themes of civic access and community support. The Team page presents factual academic project information. The content page is original writing grounded in publicly available UN and academic sources, all cited in Section 7.
  • No Data Transmission: The feedback form is handled entirely client-side. On valid submission, a confirmation message is displayed on screen and no data is sent to any external server, satisfying data minimisation principles and avoiding JANET or UK GDPR concerns.
  • Responsible Use of Assets: All team member photographs were provided by each member with explicit consent for use in this academic context. Programme directory imagery uses descriptive alt text rather than reproducing third-party copyrighted visuals without permission.
  • Academic Integrity: All SDG 16 content on the content page is original writing. No text was reproduced from external sources without citation. References are listed in Section 7 in a consistent format as required by the coursework brief.
  • No Harmful or Offensive Material: All pages are entirely educational and directly aligned with the United Nations' published goals for SDG 16. No offensive, discriminatory, misleading, or prohibited content is present on any page I authored.
