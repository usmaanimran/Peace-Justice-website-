2. Technical Description of Implemented Pages

Splash Screen

Live Page Validation

The Splash Screen was created to offer a cinematic and immersive introduction to our site. I obtained the visual depth through the application of div elements combined with personal CSS @keyframes to provide the effects of constant breathing and glowing backgrounds.

Technically, the main functionality is achieved by using an HTML tag of the form: the <meta http-equiv="refresh"> tag. I would, however, have liked to have more control for the user. I created a bespoke JavaScript code (splash.js) which runs a live visible countdown timer on the screen. I also included a skip button to force boot. When the user presses this button, my JavaScript instantly removes the timer interval and replaces the slow meta tag with window.location.href to provide an abrupt transition.

Action Impact Simulator (AIS)

Live Page Validation

The AIS is an interactive dashboard developed using pure Vanilla JavaScript. The 6 action cards are manually inbuilt into the HTML to meet the structural demands of the brief. In order to ensure smooth work by the JavaScript math, I became familiar with the custom data attributes in HTML5. I added data-pts attributes to each card in order to keep the point values.

I used a JavaScript loop to go through these cards and add event listeners. Upon clicking a card, the script will use classList.toggle('selected'), recalculate the aggregate score, and update the UI text. The dynamic circular gauge is one of the most interesting technical features I introduced here. Using the stroke-dashoffset property of an SVG circle and mathematical manipulations regarding the summation of the score, it is possible to make the gauge appearance fill up.

Content Page: Conflict to Peace

Live Page Validation

The page conforms strictly to the common global template that was created by Student 1. In this case, I was more interested in good document organization. I created the essay with semantic HTML5 markups such as <article>, <section>, and <aside> such that the browser actually comprehends the layout. I also used a Table of Contents which made use of internal anchor links (href="#section-id") that enable a user to jump directly to various sections of the essay.

3. Challenges & Lessons Learned

Challenge 1: CSS vs. Global JavaScript Conflict

There was a giant technical problem that occurred when we began to incorporate my personal pages with the template of Student 1. My CSS-only "Go to Top" button on my Content Page was the result of much time spent fitting the strict coursework brief. The back-to-top button was, however, automatically injected using the global main.js file, which was programmed to inject a JavaScript-driven button on all pages. As I loaded my page, I realized there were two overlapping buttons fighting for clicks!

Being a beginner, the initial thought that came to my mind was to simply delete the global script, but then I knew that would break the rest of my team's pages. This made me acquire knowledge on DOM conditional logic. Our solution was to change the global script to do a check before its button is created: if (!document.querySelector('.back-to-top-css')). This logic amounts to: "Create the JS button only when Usmaan's CSS button does NOT exist on this page." The value of understanding variable scope and conditional delivery in modular team applications was strongly enforced by this experience.

Challenge 2: State Management and Logic Bugs in the Simulator

In the construction of the Action Impact Simulator, a large logic bug occurred. My JavaScript had to serve as a way of adding points each time a card was clicked. A user who clicked the "Vote Locally" card five times would simply get an infinite number of points! I needed time to learn how to monitor the state of the card. I was taught how to verify whether a card was already active. With the script now depending on card.classList.toggle('selected'), it determines whether the card is being turned on or off, and then adds or subtracts the given data-pts value. It taught me that code writing is not just about making things look good, but anticipating how users will attempt to break your application.

Challenge 3: The Skip Button that Cannot be Clicked (Z-Index Mystery)

When constructing the Splash Screen, I made the background deep and cinematic with div elements stacked on top of each other. All appeared aesthetically flawless, and I had my “Force Boot” skip button exactly where I desired. But upon clicking it in the browser, nothing occurred. The hover effect was not activated, and the JavaScript event listener remained silent.

Hours of debugging allowed me to find out it was a CSS stacking problem. Since I had set my glowing overlay background to position: absolute to cover the whole screen, it was invisibly sitting on top of my button, catching all the mouse clicks. I got to know about the CSS "stacking context" and rectified the problem by assigning my main wrapper position: relative and z-index: 10. This brought the button physically over the background layers, rendering it fully clickable. It made me understand that just because you can see an element doesn't mean it's physically on top.

Challenge 4: The Jarring Background Cut (Cinematic Crossfading)

For the Action Impact Simulator, I initially decided to change the background depending on the user's score by using JavaScript to replace the background-image property on the <body> element. However, the transition was a sharp, abrupt cut that totally destroyed the movie-theater essence I was hoping to achieve. I attempted to add a CSS transition, but soon discovered that browsers have a hard time smoothly transitioning intricate background gradients.

To address this, I entirely re-engineered my approach. Rather than replacing the background of the body, I used three distinct and fixed div layers (#bgLow, #bgMed, #bgHigh), positioned them in the background with an opacity of 0. My JavaScript logic then changed the active state by adding an .active class, transitioning the opacity to 1 over 1.6 seconds. This created beautiful, smooth crossfading. It was an enormous lesson in creative problem-solving: you cannot always force a single property to do what you wish, and must sometimes build a mechanical workaround.

Challenge 5: Reacting Grids and Squished Cards

The initial 6 hardcoded action cards I constructed were built on a large desktop screen. I inserted them using simple CSS next to each other and was proud of it—until I shrunk my browser window to imitate a cell phone. The cards did not stack; they merely squeezed together until the text overlapped and became completely unreadable.

I first attempted to write a series of complex @media queries to force column formats at various screen sizes. Then, while researching, I found the strength of the auto-fit and minmax() functions in CSS Grid. I replaced dozens of code lines with a single one: grid-template-columns: repeat(auto-fit, minmax(240px, 1fr));. This commanded the browser to fit as many 240px cards in a row as it could, wrapping them automatically when the screen became too narrow. It was a breakthrough in understanding modern fluid responsive design.

4. Accessibility Considerations

As I was working on the site, I understood that it is easy to make a webpage look cool, but it does not do much good when certain individuals are not able to use it. I placed a very high value on keyboard navigation and screen-reader support:

  • Keyboard Interactivity: I had initially created my action cards using the <div> tag, which meant a user using the Tab key could not select them. I researched and added tabindex="0" and role="button" to the HTML. Moreover, I coded a specific event listener in ais.js that responds to keydowns of the 'Enter' and 'Space' keys, allowing users to play the simulator fully without a mouse. I also implemented aria-pressed="true/false" states to dynamically inform screen readers when a card is selected.
  • Semantic Context: All content images on my page have descriptive alt text to ensure clear narratives. Conversely, for the decorative SVG icons applied inside the simulator cards, I learned how to apply aria-hidden="true". This ensures the audio experience remains clean, preventing the screen reader from reading unstructured raw SVG code.
