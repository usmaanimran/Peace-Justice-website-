1. Completed Tasks & Responsibilities

As Student 4 for Team Humpty Dumpty, my core responsibility was to design and build the personalisation layer of our UN SDG 16 website, as well as provide a clear navigational overview for the whole site. I wanted users to feel a genuine connection to the Peace & Justice mission by creating a profile that reflects their own values and commitments. My deliverables were as follows:

  • A fully interactive three-step User Profile page built with prompt() dialogues, persistent localStorage storage, and a live <progress> indicator (profile.html).
  • A visual Sitemap page constructed entirely in inline SVG, presenting the full site architecture as a hierarchical node graph (sitemap.html).
  • W3C validation of both pages and production of the corresponding validation_ST4.html evidence page.
  • Authoring of this Page Editor document to document technical decisions, challenges, and academic references.

2. Technical Implementation

User Profile Page — Multi-Step Prompt Flow

The profile page is driven by a three-step data-collection flow triggered when the user clicks the Start Profile button. Each step calls a dedicated function — runStep1(), runStep2(), and runStep3() — which fire sequentially inside startProfile(). Each function uses the native Window.prompt() API [1] to collect three questions from the user, storing every answer in a shared profile object.

Collected data is immediately written to the DOM using document.getElementById() [2] and element.textContent [2], making the three profile cards visible. The entire profile object is then persisted to localStorage via a dedicated saveData() function, so the profile survives page reloads. On first load, the page attempts to restore any saved data with JSON.parse(localStorage.getItem("profile")), falling back to an empty object if nothing exists yet.

Progress Bar Implementation

After all three steps complete, updateProgress() tallies every non-null answer across all steps (maximum 9 questions), calculates a percentage, and injects both the numeric value and the visual fill into the native HTML <progress> element [3]. The progress bar is styled with ::-webkit-progress-value and ::-moz-progress-bar pseudo-elements to render a glowing cyan-to-gold gradient consistent with the site's design language.

Edit Functionality

Each profile card exposes an Edit button that calls editStep(n), which re-runs the corresponding prompt sequence and calls updateProgress() afterwards. This allows the user to revise any section without restarting the entire flow, providing a smooth and forgiving user experience.

Sitemap Page — Inline SVG Architecture Diagram

Rather than using an image file or an external library, the sitemap is built entirely as inline SVG markup [2] within a responsive viewBox="0 0 1200 750" container. Each page in the site hierarchy is represented as a <g class="node"> group containing a rounded <rect> and a centred <text> element. Relationships between pages are drawn with <line> elements connecting the midpoints of parent and child nodes.

The SVG uses CSS custom properties (var(--accent-cyan), var(--accent-purple)) from the shared style.css stylesheet to ensure visual consistency with the rest of the site. Hover states are handled via CSS class selectors on the .node group, changing the stroke colour to purple on mouse-over to provide clear interactive feedback without any JavaScript overhead.

3. Challenges & Lessons Learned

Challenge 1: prompt() Returns null, Not an Empty String

When building the profile flow, I ran into an unexpected bug during testing: if a user pressed Cancel on any prompt() dialogue [1], the value stored was the JavaScript literal null, not an empty string. This meant my initial if (name) guard correctly skipped the DOM update, but then JSON.stringify would write the string "null" into localStorage — and on the next reload, JSON.parse would restore it as the actual null value, causing the card paragraph to read "Name: null" on-screen.

I solved this by ensuring the DOM-update guard (if (name)) was always evaluated before writing to the paragraph, and by only incrementing the count variable for genuinely truthy answers. This taught me that JavaScript's loose type coercion can produce silent bugs that only surface in edge cases, and that defensive coding around user-cancellable dialogues is essential.

Challenge 2: SVG Text Not Centring Consistently Across Browsers

In my initial sitemap build, I relied solely on text-anchor="middle" to centre the node labels. On Chrome the alignment was perfect, but on Firefox the text was rendering slightly off-centre vertically. I investigated the SVG specification [2] and discovered that vertical alignment in SVG text requires the separate attribute dominant-baseline="middle" to be explicitly declared alongside text-anchor="middle". Adding both attributes resolved the cross-browser inconsistency immediately. This was a valuable reminder that SVG has its own independent layout model, separate from CSS flexbox or block formatting.

Challenge 3: The Progress Bar Jumping to 100% on Skipped Questions

An early version of updateProgress() calculated the percentage based on how many steps had been triggered rather than how many individual answers were provided. This meant that a user who pressed Cancel on all nine questions still saw a 100% filled progress bar after clicking Start Profile — a completely misleading representation of their engagement.

I refactored the logic to count individual truthy values using separate completed.step1, completed.step2, and completed.step3 integer counters, each incremented only when the corresponding answer is non-null. The total is then divided by 9 and multiplied by 100 to produce an accurate percentage. This experience reinforced how important it is to define exactly what "completion" means before writing the logic to measure it.

Challenge 4: SVG Lines Connecting to Wrong Coordinates After Repositioning Nodes

When I rearranged the sitemap hierarchy to better reflect the actual site structure, I found that all the <line> coordinates were hardcoded to the old node positions. Moving a single node required manually recalculating every x1, y1, x2, y2 attribute for every connected line. After going through this painful recalculation three times, I developed a systematic approach: I first sketched the entire layout on paper with exact x/y coordinates for each node centre before writing a single line of SVG. This meant any repositioning was planned upfront, and the SVG was written once correctly rather than patched repeatedly.

Challenge 5: localStorage Persisting Stale Data During Development

During development I repeatedly changed the structure of the profile object — renaming keys and adding new fields. Because localStorage persists across page reloads, the browser kept loading the old data structure, causing undefined values to appear in the DOM whenever the code expected a key that the saved object did not yet contain.

I learned to clear localStorage manually in DevTools between structural changes, and I also added a safe fallback in the initialisation: JSON.parse(localStorage.getItem("profile")) || {step1:{}, step2:{}, step3:{}}. This ensures that even if stored data is malformed or outdated, the app always starts from a valid empty structure. It taught me to treat persisted data as potentially untrusted and always validate or default it on load.

4. Accessibility Considerations

Building accessible pages was a priority throughout development. Below are the specific measures I implemented for both deliverables:

  • SVG Accessibility: Following W3C SVG accessibility guidelines [7], the sitemap <svg> element carries a descriptive aria-label so screen readers announce its purpose. Individual node <g> groups use role="img" and aria-label attributes to identify each page by name, ensuring a screen-reader user can navigate the site structure without visual reference. Purely decorative connector <line> elements are marked aria-hidden="true" to prevent clutter in the accessibility tree.
  • Keyboard Navigation on Profile Buttons: The Start Profile and Edit buttons are native HTML <button> elements, which are natively focusable and operable via the keyboard without any additional tabindex or role attributes. This ensures full keyboard access to the entire profile flow out of the box.
  • Progress Bar Semantics: The native <progress> element [3] exposes its value and max attributes to the accessibility tree automatically. The adjacent <p id="progressText"> provides a plain-text fallback ("Progress: 67%") that is announced to screen readers even in browsers that do not expose the progress element role correctly.
  • Colour Contrast: All text on profile cards uses var(--c-muted) against the dark glass background, maintaining a contrast ratio above the WCAG AA threshold of 4.5:1 for normal text.

5. Compliance with JANET Regulations

Both pages I authored are in full compliance with the JANET Acceptable Use Policy and the university's web publishing guidelines:

  • Content Integrity: All profile prompts and sitemap labels are original, educational content directly related to SDG 16 themes — Peace, Justice, and community engagement. No offensive, defamatory, or prohibited material is present anywhere in my pages.
  • User Data: The profile data entered by the user is stored exclusively in the user's own browser via localStorage. No data is transmitted to any external server, ensuring full compliance with data minimisation principles and protecting user privacy.
  • No Third-Party Scripts: Both pages rely only on the shared style.css and main.js files produced by the team — no external CDN libraries, tracking scripts, or advertising code is loaded, in line with acceptable use requirements.
  • Accurate Attribution: All technical techniques and external references are fully cited in Section 7 of this document using Harvard referencing, in compliance with the coursework brief [8].
